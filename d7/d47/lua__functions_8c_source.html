370 /** \defgroup LuaVolumeMesher Volume Meshers
371  * \ingroup LuaMesh
372  *
373  * ## General Concepts
374  *
375  * A volume mesher generates cells (chi_mesh::Cell) which can be
376  * either 2D polygons (chi_mesh::CellPolygon) or 3D polyhedrons
377  * (chi_mesh::CellPolyhedron). All cell objects are pushed into
378  * a vector located in a chi_mesh::MeshContinuum after creation.
379  * Right now only a single continuum can be operated on and this is normally
380  * by means of attaching a region to a solver. \n
381  *
382  * \n
383  * Right now 2D meshes can be partitioned but the full 2D cell geometry
384  * exists on each processor. This is mostly because of the way the extruder
385  * works. This means that for a 2D problem a continuum will have ALL of the
386  * cells (as fully defined cells), in every process, in the member
387  * vector "cells" of the continuum.
388  * The same can not be said for 3D extruded meshes where each process has all
389  * the nodes but if cells are not local then only placeholders are uploaded.
390  * Placeholders are basically the base class chi_mesh::Cell and contains only
391  * the cell's partition_id and its centroid. The concept that these placeholders
392  * are uploaded allows a sweeping order to figure out dependencies. A similar
393  * strategy would have to be devised for using third-party meshes.\n
394  *
395  * \n
396  * Global mesh references are maintained in chi_mesh::MeshContinuum::cells.
397  * This contains the actual mesh object. Local indices are stored in
398  * chi_mesh::MeshContinuum::local_cell_glob_indices and are the global indices
399  * of local cells. Conversely the local indices, given a global index,
400  * are stored in chi_mesh::MeshContinuum::glob_cell_local_indices.
