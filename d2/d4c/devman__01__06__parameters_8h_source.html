6\section DevManParametersSec1 1 The general philosophy of Static Registration
7
8First, what is <B>Static Registration</B>? When a c++ program is executed there
9are a number of phases that occur in order. Once such phase, before entering the
10`main` function, is the phase in which all static variables are initialized. We
11shall call this phase the static initialization phase.
12In the most basic sense static registration is the process whereby we create a
13registry of object-constructors during this phase. This registry can then used,
14during the dynamic phases of the program, to construct objects.
15A simple example is the code below:
16\code
17#include <iostream>
18#include "DoSomethingFunctions"
19
20static int a = DoSomething();
21
22int main()
23{
24 std::cout << "The value of a is: " << a << std::endl;
25}
26\endcode
27where the variable `int a` is initialized, before the `main` function starts,
28using a function call, which happens during the static initialization phase of
29the program.
30
31\subsection DevManParametersSec1_1 Rigging static initialization for object factories or object makers
32
33We can use the rather odd dynamics of singleton objects to create a static
34registry of function creation calls during static initialization. In the code
35below we define an `ObjectMaker` class that is a singleton. A singleton is an
36object of which only a single instance can be created, which we do by declaring
37a static instance, a bunch of static members methods, a private constructor and
38deleted copy- and move constructors.
39
40The `ObjectMaker` maintains a map of string keys to object construction calls.
41The construction calls always return a pointer to a base class `Object` but can
42construct any child object by using the shown template arguments. In order for
43us to achieve basic static registration we need two templated-methods, these
44are `CallObjectConstructor` and `RegisterObject` in the code below.
45The first method defines a function-address that we can put into our registry.
46The second method is the actual method we call to make a registry entry. When we
47then want to make a registered object we simply call the `MakeRegisteredObject`
48method.
49
50\code
51#include <iostream>
52#include <map>
53#include <functional>
54
55class Object
56{
57public:
58 int member_;
59 Object() : member_(99) {}
60 Object(int member) : member_(member) {}
61};
62
63class ChildObject : public Object
64{
65public:
66 ChildObject() : Object(999) {}
67};
68
69//===============================================
70typedef std::function<Object*()> CreationFunction;
71class ObjectMaker // Singleton
72{
73public:
74 std::map<std::string, CreationFunction> registry_;
75
76public:
77 static ObjectMaker& GetInstance() noexcept
78 {
79 static ObjectMaker singleton;
80 return singleton;
81 }
82 template <typename T>
83 static Object* CallObjectConstructor() { return new T(); }
84
85 template <typename T>
86 static int RegisterObject(const std::string& register_name)
87 {
88 auto& instance = GetInstance();
89 instance.registry_.insert(
90 std::make_pair(register_name, &CallObjectConstructor<T>));
91 return 0;
92 }
93
94 static Object* MakeRegisteredObject(const std::string& obj_name)
95 {
96 auto& instance = GetInstance();
97
98 return instance.registry_.at(obj_name)();
99 }
100private:
101 ObjectMaker() = default;
102 ObjectMaker(const ObjectMaker&) = delete;
103 ObjectMaker(const ObjectMaker&&) = delete;
104 ObjectMaker& operator=(const ObjectMaker&) = delete;
105};
106
107//===============================================
108// This is where we statically register an object
109static int abc12345 = ObjectMaker::RegisterObject<Object>("Parent");
110static int abc12346 = ObjectMaker::RegisterObject<ChildObject>("Child");
111
112
113int main()
114{
115 Object objectA(98); // Conventional object creation
116 auto objectB = ObjectMaker::MakeRegisteredObject("Parent");
117 auto objectC = ObjectMaker::MakeRegisteredObject("Child");
118
119 std::cout << "Object A.member_ = " << objectA.member_ << "\n";
120 std::cout << "Object B.member_ = " << objectB->member_ << "\n";
121 std::cout << "Object C.member_ = " << objectC->member_ << "\n";
122
123 return 0;
124}
125\endcode
126Output:
127\verbatim
128Object A.member_ = 98
129Object B.member_ = 99
130Object C.member_ = 999
131\endverbatim
132
133Notice that we abuse the static initialization phase right above the `main`
134function. We call the static singleton `ObjectMaker` and `RegisterObject` to
135create registry entries for a `Parent` and a `Child`. In the main function we
136then construct some of these statically registered objects.

138\subsection DevManParametersSec1_2 Implementation in ChiTech
139In ChiTech we have to, of course, enhance the functionality by a lot. Firstly,
140we ought not to use raw pointers, therefore we use shared pointers
141to a base class object for which we use the base class `ChiObject` for all
142objects that can be statically registered.
143Secondly, it is very rare for an object to be created with no parameters
144(i.e., empty constructors). We developed a separate system to
145control object construction parameters (more on this later). Additionally,
146we do not want developers to come up with unique dummy variable names, i.e.,
147`static int abc12345` or `static int abc12346`, to make use of the
148`ChiObjectFactory`.
149
150In order to register an object within the `ChiObjectFactory` we created a macro
151called `RegisterChiObject(namespace_name, obj_name)`. The arguments to this
152macro, `namespace_name` and `obj_name`, then get used to construct the following
153\code
154RegisterChiObject(namespace_name, ObjectName);
155//
156// becomes:
157//
158static char unique_var_name_object_ObjectName_0 =
159 ChiObjectFactory::AddObjectToRegistry<ObjectName, ChiObject>("namespace_name",
160 "ObjectName")
161\endcode
162which allows us to abuse static initialization to register the object with
163text-name `"namespace_name::ObjectName"`.
164
165We also implemented a number of "tricks". For example we created a
166behind-the-scenes connection to the lua console so that registered objects can
167be created from the lua input system. We also created a "RegistryDump" where
168each registered object can dump its input arguments specification so we can use
169it to build documentation (lessening the burden on the developer).

171\section DevManParametersSec2 2 The InputParameters system
172All statically registered objects have a common input parameters interface
173implemented in the general class `chi_objects::InputParameters`. This allows
174us to fully support static registration if an object has the following basic
175structure
176\code
177#include "ChiObject/chi_object.h"
178
179namespace zorba
180{
181
182class CoolObject : public ChiObject
183{
184 const int a_;
185 const double b_;
186 //Misc. members
187public:
188 static chi_objects::InputParameters GetInputParameters();
189 CoolObject(const chi_objects::InputParameters& params);
190
191 //Misc. methods
192};
193
194}
195\endcode
196Here we have the static member method `GetInputParameters` which returns a
197`chi_objects::InputParameters` object used to assign parameters from user input,
198and the constructor `CoolObject(const chi_objects::InputParameters& params);`
199which uses such an object.
200
201In the definition (`.cc`) file we then have
202\code
203#include "CoolObject.h"
204
205#include "ChiObject/object_maker.h"
206
207namespace zorba
208{
209
210RegisterChiObject(chi_unit_tests, CoolObject);
211
212chi_objects::InputParameters CoolObject::GetInputParameters()
213{
214 chi_objects::InputParameters params = ChiObject::GetInputParameters();
215
216 params.AddRequiredParameter<int>("a", "Coolness factor a");
217 params.AddOptionalParameter("b", 1.0, "Coolness factor b");
218
219 return params;
220}
221
222CoolObject::CoolObject(const chi_objects::InputParameters& params) :
223 ChiObject(params),
224 a_(params.GetParamValue<int>("a")),
225 b_(params.GetParamValue<double>("b"))
226{
227}
228
229}//namespace zorba
230\endcode
231
232Creating this object from lua is then as easy as
233\code
234params = {a = 98, b = 99.0}
235cool_obj = zorba.CoolObject.Create(params)
236\endcode
