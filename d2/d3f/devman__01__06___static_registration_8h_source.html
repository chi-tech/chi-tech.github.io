8\section DevManParametersSec1 1 The general philosophy of Static Registration
9
10First, what is <B>Static Registration</B>? When a c++ program is executed there
11are a number of phases that occur in order. Once such phase, before entering the
12`main` function, is the phase in which all static variables are initialized. We
13shall call this phase the static initialization phase.
14In the most basic sense static registration is the process whereby we create a
15registry of object-constructors during this phase. This registry can then used,
16during the dynamic phases of the program, to construct objects.
17A simple example is the code below:
18\code
19#include <iostream>
20#include "DoSomethingFunctions"
21
22static int a = DoSomething();
23
24int main()
25{
26 std::cout << "The value of a is: " << a << std::endl;
27}
28\endcode
29where the variable `int a` is initialized, before the `main` function starts,
30using a function call, which happens during the static initialization phase of
31the program.
32
33\subsection DevManParametersSec1_1 Rigging static initialization for object factories or object makers
34
35We can use the rather odd dynamics of singleton objects to create a static
36registry of function creation calls during static initialization. In the code
37below we define an `ObjectMaker` class that is a singleton. A singleton is an
38object of which only a single instance can be created, which we do by declaring
39a static instance, a bunch of static members methods, a private constructor and
40deleted copy- and move constructors.
41
42The `ObjectMaker` maintains a map of string keys to object construction calls.
43The construction calls always return a pointer to a base class `Object` but can
44construct any child object by using the shown template arguments. In order for
45us to achieve basic static registration we need two templated-methods, these
46are `CallObjectConstructor` and `RegisterObject` in the code below.
47The first method defines a function-address that we can put into our registry.
48The second method is the actual method we call to make a registry entry. When we
49then want to make a registered object we simply call the `MakeRegisteredObject`
50method.
51
52\code
53#include <iostream>
54#include <map>
55#include <functional>
56
57class Object
58{
59public:
60 int member_;
61 Object() : member_(99) {}
62 Object(int member) : member_(member) {}
63};
64
65class ChildObject : public Object
66{
67public:
68 ChildObject() : Object(999) {}
69};
70
71//===============================================
72typedef std::function<Object*()> CreationFunction;
73class ObjectMaker // Singleton
74{
75public:
76 std::map<std::string, CreationFunction> registry_;
77
78public:
79 static ObjectMaker& GetInstance() noexcept
80 {
81 static ObjectMaker singleton;
82 return singleton;
83 }
84 template <typename T>
85 static Object* CallObjectConstructor() { return new T(); }
86
87 template <typename T>
88 static int RegisterObject(const std::string& register_name)
89 {
90 auto& instance = GetInstance();
91 instance.registry_.insert(
92 std::make_pair(register_name, &CallObjectConstructor<T>));
93 return 0;
94 }
95
96 static Object* MakeRegisteredObject(const std::string& obj_name)
97 {
98 auto& instance = GetInstance();
99
100 return instance.registry_.at(obj_name)();
101 }
102private:
103 ObjectMaker() = default;
104 ObjectMaker(const ObjectMaker&) = delete;
105 ObjectMaker(const ObjectMaker&&) = delete;
106 ObjectMaker& operator=(const ObjectMaker&) = delete;
107};
108
109//===============================================
110// This is where we statically register an object
111static int abc12345 = ObjectMaker::RegisterObject<Object>("Parent");
112static int abc12346 = ObjectMaker::RegisterObject<ChildObject>("Child");
113
114
115int main()
116{
117 Object objectA(98); // Conventional object creation
118 auto objectB = ObjectMaker::MakeRegisteredObject("Parent");
119 auto objectC = ObjectMaker::MakeRegisteredObject("Child");
120
121 std::cout << "Object A.member_ = " << objectA.member_ << "\n";
122 std::cout << "Object B.member_ = " << objectB->member_ << "\n";
123 std::cout << "Object C.member_ = " << objectC->member_ << "\n";
124
125 return 0;
126}
127\endcode
128Output:
129\verbatim
130Object A.member_ = 98
131Object B.member_ = 99
132Object C.member_ = 999
133\endverbatim
134
135Notice that we abuse the static initialization phase right above the `main`
136function. We call the static singleton `ObjectMaker` and `RegisterObject` to
137create registry entries for a `Parent` and a `Child`. In the main function we
138then construct some of these statically registered objects.
139
140\subsection DevManParametersSec1_2 Implementation in ChiTech
141In ChiTech we have to, of course, enhance the functionality by a lot. Firstly,
142we ought not to use raw pointers, therefore we use shared pointers
143to a base class object for which we use the base class `ChiObject` for all
144objects that can be statically registered.
145Secondly, it is very rare for an object to be created with no parameters
146(i.e., empty constructors). We developed a separate system to
147control object construction parameters (more on this later). Additionally,
148we do not want developers to come up with unique dummy variable names, i.e.,
149`static int abc12345` or `static int abc12346`, to make use of the
150`ChiObjectMaker`.
151
152In order to register an object within the `ChiObjectMaker` we created a macro
153called `RegisterChiObject(namespace_name, obj_name)`. The arguments to this
154macro, `namespace_name` and `obj_name`, then get used to construct the following
155\code
156RegisterChiObject(namespace_name, ObjectName);
157//
158// becomes:
159//
160static char unique_var_name_object_ObjectName_0 =
161 ChiObjectMaker::AddObjectToRegistry<ObjectName, ChiObject>("namespace_name",
162 "ObjectName")
163\endcode
164which allows us to abuse static initialization to register the object with
165text-name `"namespace_name::ObjectName"`.
166
167We also implemented a number of "tricks". For example we created a
168behind-the-scenes connection to the lua console so that registered objects can
169be created from the lua input system. We also created a "RegistryDump" where
170each registered object can dump its input arguments specification so we can use
171it to build documentation (lessening the burden on the developer).

173\section DevManParametersSec2 2 The InputParameters system
174All statically registered objects have a common input parameters interface
175implemented in the general class `chi_objects::InputParameters`. This allows
176us to fully support static registration if an object has the following basic
177structure
178\code
179#include "ChiObject/chi_object.h"
180
181namespace zorba
182{
183
184class CoolObject : public ChiObject
185{
186 const int a_;
187 const double b_;
188 //Misc. members
189public:
190 static chi_objects::InputParameters GetInputParameters();
191 CoolObject(const chi_objects::InputParameters& params);
192
193 //Misc. methods
194};
195
196}
197\endcode
198Here we have the static member method `GetInputParameters` which returns a
199`chi_objects::InputParameters` object used to assign parameters from user input,
200and the constructor `CoolObject(const chi_objects::InputParameters& params);`
201which uses such an object.
202
203In the definition (`.cc`) file we then have
204\code
205#include "CoolObject.h"
206
207#include "ChiObject/object_maker.h"
208
209namespace zorba
210{
211
212RegisterChiObject(chi_unit_tests, CoolObject);
213
214chi_objects::InputParameters CoolObject::GetInputParameters()
215{
216 chi_objects::InputParameters params = ChiObject::GetInputParameters();
217
218 params.AddRequiredParameter<int>("a", "Coolness factor a");
219 params.AddOptionalParameter("b", 1.0, "Coolness factor b");
220
221 return params;
222}
223
224CoolObject::CoolObject(const chi_objects::InputParameters& params) :
225 ChiObject(params),
226 a_(params.GetParamValue<int>("a")),
227 b_(params.GetParamValue<double>("b"))
228{
229}
230
231}//namespace zorba
232\endcode
233
234Creating this object from lua is then as easy as
235\code
236params = {a = 98, b = 99.0}
237cool_obj = zorba.CoolObject.Create(params)
238\endcode

240\subsection DevManParametersSecFAQ1 FAQ1 Documentation-strings (Doc-strings)
241All parameters must have a documentation string (short name "doc string").
242For example:
243\code
244params.AddRequiredParameter<int>("a", "Coolness factor a");
245params.AddOptionalParameter("b", 1.0, "Coolness factor b");
246\endcode
247here "Coolness factor a" and "Coolness factor b" are the doc strings for
248parameters "a" and "b" respectively.
249
250You can also add documentation for the entire object. When the documentation
251is auto-generated this description will appear at the top.
252\code
253params.SetGeneralDescription("General test object");
254\endcode
255
256By default the object's documentation will be posted to a group that is
257reference-able via doxygen as `<namespace_name>__<ObjectName>` (note the double
258underscore is a replacement for the namespace scope clarifiers `::`). You can
259change this behavior by adding doxygen code to `SetGeneralDescription`. For
260example the code below
261\code
262params.SetGeneralDescription(
263"\\defgroup lbs__OptionsBlock lbs.OptionsBlock \n"
264"\\ingroup LuaLBS\n"
265"Set options from a large list of parameters");
266\endcode
267will set the documentation to the group `lbs__OptionsBlock`, visible as
268`"lbs.OptionsBlock"` and will be grouped under the group `LuaLBS`.
269
270This functionality allows you to customize your documentation according to your
271needs.
272
273<B>Note:</B> The block of text supplied to `SetGeneralDescription` is pasted
274into a documentation page that will be processed by doxygen so all doxygen
275commands will work here (including formulas, code blocks, and links).

277\subsection DevManParametersSecFAQ2 FAQ2 Optional Parameters
278Optional parameters must always have a default value. However, you do not
279necessarily have to use it. In cases where you use a parameter only when
280supplied,
281the `InputParameters` object has the interface method `ParametersAtAssignment`
282which returns the parameters used at assignment. It is used as follows:
283\code
284CoolObject::CoolObject(const chi_objects::InputParameters& params) :
285 ChiObject(params),
286 a_(params.GetParamValue<int>("a")),
287 b_(0.0)
288{
289 const auto& user_params = params.ParametersAtAssignment();
290 if (user_params.Has("b"))
291 b_ = user_params.GetParamValue<double>("b");
292}
293\endcode

295\subsection DevManParametersSecFAQ3 FAQ3 Inheriting parameters from parent objects
296Input parameters can be inherited from parents in the `GetInputParameters`
297method call. For example
298\code
299chi_objects::InputParameters LBSSolver::GetInputParameters()
300{
301 chi_objects::InputParameters params =
302 chi_physics::Solver::GetInputParameters();
303 //
304 //etc.
305\endcode
306Gets all the parameters from the parent object `chi_physics::Solver` allowing
307us to change or add to these parameters.
308
309In fact, a pretty slick move here is that we are not bound by classical
310inheritance. The input parameters of any statically registered object can be
311snatched and used by simply calling its static `GetInputParameters` method.
312
313\subsection DevManParametersSecFAQ4 FAQ4 Changing inherited options to optional/required
314If a required parameter is to be changed to optional we can do this using the
315interface `ChangeExistingParamToOptional`, e.g.,
316\code
317chi_objects::InputParameters LBSSolver::GetInputParameters()
318{
319 chi_objects::InputParameters params =
320 chi_physics::Solver::GetInputParameters();
321
322 params.ChangeExistingParamToOptional("name", "LBSDatablock");
323\endcode
324
325Vice versa there is also `ChangeExistingParamToRequired<type>("name")`.

327\subsection DevManParametersSecFAQ5 FAQ5 Deprecated parameters
328Parameters in an `InputParameters` block can be marked as deprecated via 3
329different mechanisms:
330- Marking them with a deprecation warning `MarkParamaterDeprecatedWarning`. This
331 will generate just a warning if the parameter is supplied..
332- Marking them with a deprecation error `MarkParamaterDeprecatedError`. This
333 will generate an error if the parameter is supplied..
334- Marking them with as renamed `MarkParamaterRenamed`. This will generate an
335 error if the parameter is supplied if the parameter is supplied.
336
337\subsection DevManParametersSecFAQ6 FAQ6 Constrained ranges
338Parameters in an input parameters block can be constrained to the following
339- `chi_data_types::AllowableRangeList`, created with a `std::vector` of a
340 specific type. The parameter can then only be one of those types.
341- `chi_data_types::AllowableRangeLowLimit`. Parameter must be >= than the
342 value specified when the range is specified as closed or just > that the value
343 specified when the range is specified as open.
344- `chi_data_types::AllowableRangeHighLimit` same as the low limit but just with
345 a smaller than comparison.
346- `chi_data_types::AllowableRangeLowHighLimit` combination of both a high and
347 a low limit range.
