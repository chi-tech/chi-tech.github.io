10 \subsection devman_introduction_0 Object Oriented Programming or Functional Programming
11 
12 ### <CENTER>Chi-Tech uses a healthy balance between OOP and FP</CENTER>
13 
14 The fundamental language with which a human provides instructions to a
15  computer has always been <B>assembly language</B>. This language,
16  which was pioneered halfway through the 20th century, is specific to
17  a computer architecture and became the lowest level of programming that
18  rose above simply supplying bytes of binary code to the computer.
19 
20 Assembly language comprises combinations of keywords and/or numbers to
21  form instructions, with each instruction telling the processor what
22  action to take, whether it is to move data from one memory location
23  to another or to perform an arithmetic operation on some data. By
24  stringing together numerous instructions the complexity of a human
25  thought-process could be made repeatable by a computer, however, the
26  process of generating assembly language programs quickly became tedious.
27  It did not take long before mankind comprehended the need for ``packaging"
28  multiple instructions into <B>macros</B>, which could be reused and
29  hence the modern programming language was born where macros are
30  standardized and collected into higher level languages
31  (i.e., FORTRAN, C, etc).
32 
33 
34 In the 1950s scientific programming revolved around
35  <B>block programming</B>, a programming paradigm where
36  thought-processes are arranged as blocks that creates-,
37  transforms- or uses data. This era of programming also included
38  the use of the famous "GOTO" statement which led to the concept
39  of what we today call "spaghetti-code", a style of programming
40  only seen in older programs. The biggest concern with this style
41  is the amount of effort required by a person, other than the creator
42  of the programmer, to understand the overall program structure and flow.
43  The difficulties are mainly associated with the limitations of the
44  human mind to maintain scope and in many cases programmers would find
45  themselves having trouble comprehending programs that they wrote a
46  few weeks or months ago. These problems were greatly reduced with
47  the advent of <B>structured programming</B> where the flow of the
48  program is controlled by means of control blocks like if/then/else
49  statements within which blocks of code, called "functions", could
50  be executed. This aligned well with the industrial era where procedures
51  could be regarded as stand-alone parts and the overall flow of the program
52  could be visualized by a process diagram. The general term used
53  today is <B>functional programming</B> (FP) and is very effective
54  for certain algorithms.
55 
56 
57 The general motivation for functional programming stemmed from a
58  development point of view. In the true sense of talking to the
59  computer, the FP paradigm still resulted in sets of instructions
60  to the computer, however, it was found that reusing, changing or
61  understanding code using block programming proved to be a tremendously
62  difficult task, especially when "GOTO" statements constantly interrupted
63  the linear flow of a program. Functional programming overcame this
64  difficulty by defining stand-alone procedures that had predefined inputs
65  and outputs along with a concept called <B>local scope</B>, which is
66  a tremendous improvement that allowed much flexibility with variable
67  names and allowed the programmer to remove the dependence on global data.
68  This paradigm is still very much valid today and greatly improved the
69  process of code development.
70 
71 
72 Another programming paradigm, called <B>object oriented programming</B>
73  (OOP), was created in the 1960s to allow programmers to structure
74  their code as objects that interact with each other. It ushered-in
75  an era where the complexity of a computer programming could be limitless
76  and the ease of comprehension of computer programs could be increased
77  by relating code concepts to real world objects. This paradigm grew
78  very popular and successful. It allowed for an alternative to the
79  control blocks used in procedural programming and allowed programs to
80  reduce or even eliminate cache-misses (a performance aspect). It's
81  success however, was not long-lived, as the difficulties associated
82  with this paradigm became apparent. One such difficulty was that a
83  program's evolution became dependent on the underlying architecture/layout
84  of objects. If the architecture was flawed from the start the program's
85  development would eventually grind to a halt, where additions or
86  modifications became near impossible.
87 
88 OOP allows for the implementation of advanced concepts some of them
89  plagued with tremendously detrimental risk. Concepts such as inheritance
90  and encapsulation offered many rewards whilst simultaneously introducing
91  problems.
92 
93 <B>Inheritance</B> allows one to define a parent class of objects,
94  with associated procedures called methods, that provide basic
95  functionality that can be used or specialized by child classes.
96  A typical example is a parent class "solver" specialized
97  into "diffusion-solver" or "transport-solver". The concept is
98  useful in many circumstances and is generally very successful if
99  not for the risk of creating categorical hierarchies that ultimately
100  become unmanageable. Inheritance ultimately allows programmers
101  to "program" their code into a corner, from which there is no escape
102  other than to refactor large portions of
103  architecture (if not the whole architecture). In scientific programs,
104  that need to solve a multitude of different kinds of problems,
105  inheritance can create such a large hierarchical depth that the
106  program becomes very dependent on the programmer's knowledge of
107  the underlying architecture and less on the actual knowledge of
108  problem-solving techniques. Therefore, when the program grows beyond
109  a certain level of complexity, it can be more efficient to write
110  specialized code from scratch than to devote the time to learn and
111  comprehend the hierarchy. Inheritance is not dispensable however,
112  and has a very unique place in modern programming. The ultimate
113  solution is to keep scientific program to a very shallow level of
114  inheritance in order to minimize the hierarchical depth of the
115  architecture.
116 
117 <B>Encapsulation</B> allows certain data items of an object to
118  be <B>private</B> and therefore protected from incorrect
119  manipulation. It has provided great utility and will be a key
120  feature for many applications but unfortunately the concept also
121  fuels the over-engineering of data structures. This is because
122  objects that are over-encapsulated, cannot easily be modified by
123  procedures living in a FP paradigm, reducing the compatibility of
124  FP with OOP. Architectures that require perfect protection through
125  encapsulation often exhibit an overwhelming focus on the actual
126  "encapsulation" and less focus on the actual working code. There is
127  a healthy trade-off between creating accessor/mutator
128  functions (i.e., getters/setters) and how likely it is that unprotected
129  access will cause undetected problems. In most cases scientific
130  programmers should be more reliant on unit-tests as compared to
131  encapsulation.
132 \n
133 \n
134 The modern focus of scientific programs should be a healthy balance
135  between OOP and FP and therefore Chi-Tech follows such a paradigm.
