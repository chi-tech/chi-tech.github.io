3 \section sec0 Coding conventions
4
5 In general we follow the
6 <a href="https://google.github.io/styleguide/cppguide.html">Google C++ style guide</a>.
7 We now will dictate a few specific items. Refer to the image below for different
8 text case-namings:
9 \image html "Cases.png" "Different case names and how they look" width=220px
10 \n\n
11 Let us say it here explicitly, <B>all variable-, function-, method- names should be descriptive</B>,
12 avoid abbreviation, rather sacrifice horizontal space for readability.
13
14 \subsection devman1_sec0_1 File names
15
16 - <B>Filenames</B> should be all lower case and may include "_" underscores. i.e.
17 chi_tech_main.cc. snake_case is preferred.
18 - Do not uses dashes `-` in file names
19 - Do not uses dashes spaces in file names (Them Windows users!)
20 - Do not use filenames that already exist in `/usr/include`
21 - In general, make your filenames very specific. For example, use
22 `http_server_logs.h` rather than `logs.h`. A very common case is to have a pair
23 of files called, e.g., `foo_bar.h` and `foo_bar.cc`, defining a
24 class called `FooBar`
25 - Header files have a `.h` extension, c/c++ code files have a `.cc` extension.
26 - Classes and Categorical functions should be contained in their own folder
27 unless they are simple.
28 - Folder depth shall be kept to a minimum and not exceed a depth of 3
29 sub-levels unless very well justified, i.e. `ChiMath/Discretization/PWL`
30 - Within a class folder the class declaration must be contained in its own
31 header file. Exceptions to this are allowed for templates and in-line
32 functions
33 - When the member methods/functions become very large, consider functionally
34 splitting them. I.e. use:
35 - classname_00_constrdestr.cc for the constructor and destructor
36 - classname_01_init.cc for an initialize call
37 - classname_02_ops.cc for numerous small operations.
38
39\image html "DevMan_Filenames.png" "Example 1" width=300px
40
41 - If the class is really small then it should have a header file and an
42 associated .cc file
43
44 \image html "DevMan_Filenames1.png" "Example 2" width=350px

46\subsection devman1_sec0_2 General code
47
48 - <B>Variables</B> use snake_case, must use lower case and include "_" between words. i.e.
49 num_faces. Variable must not start or end with an underscore.
50 - <B>class Members</B> same as variables but with a trailing "_". `struct`
51 members do not need trailing underscores. See more about this below.
52 - <B>Namespaces</B> use snake_case, should be lower case and may include underscores
53 - <B>FunctionNames</B> use PascalCase, have no underscores and each word starts with a capital
54 letter. Underscores can be allowed in some circumstances.
55 - <B>Classes, structures, and types</B> use PascalCase, have each
56 word start with a capital letter. For example chi_mesh::Vector is a structure
57 called Vector and is defined in the namespace chi_mesh.
58 - <B>enums and constants</B> use SCREAMING_SNAKE_CASE.
59
60\code
61int good_variable_name; // Good variable name
62int bvm; // Bad variable name
63int DontUseThisName; // Do not name variables with capital
64
65void ThisIsAFunctionName(int parameter1)
66{
67 return;
68}
69
70class CoolDataStructure // Standalone class declaration
71{
72public:
73 int data;
74};
75
76namespace my_space // Namespace definition
77{
78 class CoolerDataStructure;
79};
80
81
82class my_space::CoolerDataStructure
83{
84public:
85 int data;
86};
87
88class my_space::CoolerDataStructure {
89public:
90 int data;
91};
92\endcode

94## More on Class member conventions
95- All class members should be named the same as regular variables, but with
96 a trailing underscore ('_'), e.g., `cell.centroid_`.
97- `struct` members should NOT be followed by an underscore.
98- Strongly consider making all class members `private/protected` unless
99 - They are `const`.
100 - They cause a sensitivity to speed, e.g., `chi_mesh::cell` does not have
101 accessor functions because `cell.vertex_ids_` is faster than
102 `cell.GetVertexIDs()`
103- References to `private/protected` members should be obtained with a method
104 call. A recommended convention is starting the method call with `Get`, e.g.,
105 `object.member` should be referenced with
106 `object.GetMember()`. This is only guidance. It is perfectly fine to have
107 `object.Member()` or anything that makes sense for the given use case.
108- Use `const` and non-`const` getters judicially. Favor using setters over
109 non-const reference if possible, e.g., setting a single value can be done
110 with `GetSingleValue`/`SetSingleValue`, however, vector manipulation might
111 be better done with `GetVector` followed by manipulations.

113\subsection devman1_sec0_3 Tabs, spaces and braces
114
115In order to save horizontal space standard indents are to be 2 spaces instead
116of 4 space equivalent tabs. Other than this the convention is flexible.\n\n
117
118Curly braces, parentheses and block braces does not have a specific requirement
119other than being used in a sense that is optimal with respect to readibility.\n\n
120
121Generally we require that code span a maximum of 80 characters. This is not
122a hard rule but will greatly enhance code reliability. Especially in
123split-screen configurations.

159 \section devman1_sec1 Header files
160
161 - Avoid using header files for code definition. Use only for
162 struct/class/function declaration
163 - Use <B>#ifdef</B> guards on all header files
164 - Avoid using forward declarations. Use header files instead.
165 - Maintain a consistent order of include files.
166
167 \subsection devman1_sec1_1 General rules
168 Header files are primarily used to make function/class/struct declarations
169 available to other compile units without the need to compile the associated
170 definitions. Unless you have a good reason to do so, do not define code within
171 header files. It results in extra compile times and is considered not good
172 practice. Of course there are exceptions like templating but in general code
173 definitions should be contained in .cc files, <B>header files are for code
174 declaration only</B>.
175
176 \subsection devman1_sec1_2 The #ifndef guard
177 All header files should have #define guards to prevent multiple inclusion.
178 The format of the symbol name should be _<CATEGORY>_<FILE>_H_.
179 To guarantee uniqueness, they should be based on the full path in a project's
180 source tree. For example, the file foo/src/bar/baz.h in project foo should
181 have the following guard:
182
183 \code
184#ifndef FOO_BAR_BAZ_H_
185#define FOO_BAR_BAZ_H_
186
187...
188
189#endif // FOO_BAR_BAZ_H_
190 \endcode
191
192 \subsection devman1_sec1_3 Forward declarations
193 Avoid using forward declarations where possible.
194 Just #include the headers you need.

197\section devman1_sec2 Namespace hell and typedefinitions
198
199<B>Don't use the "using namespace" directive</B>. Rather rename
200groups of namespaces. Use "typedef" where appropriate.
201
202
203\code
204// Forbidden -- This pollutes the namespace.
205using namespace foo;
206
207// Shorten access to some commonly used names in .cc files.
208namespace baz = ::foo::bar::baz;
209
210// Type definition for each cell there is a pair, number of faces and
211// face areas for each face
212typedef std::vector<std::pair<int, std::vector<double>>> CellFaceAreas;
213
214CellFaceAreas cell_face_areas;
215\endcode

220\subsection devman1_sec3_0 Line length
221
222Each line of text in your code should be at most 80 characters long.
223A line may exceed 80 characters if it is
224
225- a comment line which is not feasible to split without harming
226 readability, ease of cut and paste or auto-linking -- e.g.
227 if a line contains an example command or a literal URL longer
228 than 80 characters.
229- a raw-string literal with content that exceeds 80 characters.
230 Except for test code, such literals should appear near
231 the top of a file.
232- an include statement.
233- a header guard
234

236\subsection devman1_sec3_1 Boolean expressions
237
238When you have a boolean expression that is longer than the
239 standard line length, be consistent in how you break up the lines.
240
241In this example, the logical AND operator is always at the end
242 of the lines:
243
244\code
245if (this_one_thing > this_other_thing &&
246 a_third_thing == a_fourth_thing &&
247 yet_another && last_one) {
248 ...
249}
250\endcode

252\subsection devman1_sec3_2 Constructor Initializer lists
253
254Constructor initializer lists can be all on one line or with
255 subsequent lines indented four spaces.
256
257The acceptable formats for initializer lists are:
258
259\code
260// When everything fits on one line:
261MyClass::MyClass(int var) : some_var_(var) {
262 DoSomething();
263}
264
265// If the signature and initializer list are not all on one line,
266// you must wrap before the colon and indent 4 spaces:
267MyClass::MyClass(int var)
268 : some_var_(var), some_other_var_(var + 1) {
269 DoSomething();
270}
271
272// When the list spans multiple lines, put each member on its own line
273// and align them:
274MyClass::MyClass(int var)
275 : some_var_(var), // 4 space indent
276 some_other_var_(var + 1) { // lined up
277 DoSomething();
278}
279
280// As with any other code block, the close curly can be on the same
281// line as the open curly, if it fits.
282MyClass::MyClass(int var)
283 : some_var_(var) {}
284\endcode

286\subsection devman1_sec3_3 Operators
287
288Spacing between operators is flexible.
289
290\code
291// Assignment operators always have spaces around them.
292x = 0;
293
294// Other binary operators usually have spaces around them, but it's
295// OK to remove spaces around factors. Parentheses should have no
296// internal padding.
297v = w * x + y / z;
298v = w*x + y/z;
299v = w * (x + z);
300
301// No spaces separating unary operators and their arguments.
302x = -5;
303++x;
304if (x && !y)
305 ...
306\endcode

320\subsection devman1_sec3_5 Casting
321
322In general there are three variants of casting; C-style casting, `static_cast`
323and `dynamic_cast`.
324
325\code
326double aa = 10.5;
327
328float a = (float)aa; // C-style (Use only for numerics)
329float b = static_cast<float>(aa); // static_cast
330float c = dynamic_cast<float>(aa); // dynamic_cast
331\endcode
332
333The difference between the 3 are as follows: C-style casting is a brute-force
334way of casting one type into another. In most, if not all cases, the compiler
335will not provide any protection against casts that will be illegal. The better
336alternative is to use `static_cast` which will allow the compiler to determine
337whether the cast would be legal in a broad scope sense (i.e. casting parent
338pointer to child pointer). If the user needs to check for a valid cast, it is
339often better to use `dynamic_cast` which will return null if the cast is
340illegal, however, be aware that this uses RTTI and therefore has some cost.
341The following conventions therefore apply:
342
343- Use C-style casting only for numerical values
344- Objects and structures should use either `static_cast` or `dynamic_cast`
